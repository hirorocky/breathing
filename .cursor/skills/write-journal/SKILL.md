# Journal を書く

存在しやすい空間の探求ループにおける **観察** と **気づき** を `docs/journal/` に残す。

詳細手順・git 対応: [docs/journal/README.md](../../../docs/journal/README.md)

## 最重要: 対話してから書く

**観察** と **気づき** は、ユーザーとの対話で得た内容だけを書く。勝手に書かない。

| してよい | してはいけない |
|---|---|
| ユーザーの言葉をそのまま（最小限の整形のみ）残す | 実装差分・会話履歴から観察を推測して書く |
| ユーザーの口述を聞き、足りなければ質問する | 気づきをエージェントが先に結論づける |
| 書く前に草案を見せ、確認を取る | `/write-journal` だけで journal 全文を一気に生成する |
| ユーザーが「これでよい」と言ってからファイルを更新する | ユーザー未確認のまま `docs/journal/*.md` をコミットする |

`## 観察` と `## 気づき` が空でもよい。対話が終わるまで、推測で埋めない。

---

## 対話の流れ

### 1. 版と文脈を確認する

- どの版か（`vX.Y.Z`）。未決なら一緒に決める
- 観察期間・本番 URL（ユーザーが言うまで推測しない）
- 必要なら、コミットや diff から **事実だけ**（「何を試したか」）を箇条書きで提示し、**観察ではない** と明示する

### 2. 観察を聞く

ユーザーに聞く例:

- その版で、どんな体験だったか（質感・迷い・寂しさ・操作の重さ）
- 想定外だったこと、違和感、呼吸や滞在への影響
- 風や光を外した、など **ユーザーが口にした判断** があればそのまま残す

ユーザーがまだ観察していない場合は、journal を書かず「観察してからまた声をかけてください」と伝える。

### 3. 気づきを一緒に言語化する

- 観察のあと、「こう読めるかもしれないが、どう感じたか」を **提案** する程度にとどめる
- 気づきとしてファイルに入れるのは、ユーザーが同意・修正・言い換えしたあとの文だけ
- 番号付きの結論は、ユーザーと合意したものだけ

### 4. 次版への問い・次に試すを聞く

- ユーザーが挙げた問い・仮説を書く（エージェントがタスク化しすぎない）
- 「次に試す」は 1〜3 件。ユーザーと絞る

### 5. 草案を見せてから書き込む

- チャットに `## 観察` `## 気づき` の草案を出し、修正を受ける
- 確定後にだけ `docs/journal/vX.Y.Z.md` を新規作成または更新する

---

## 原則

1. **journal 依頼では実装を変えない**（`components/` `app/` `worker/` 等）。ユーザーが明示したときだけコードへ手を伸ばす
2. **存在負荷**の観点で書く（下がったか、上がったか、副作用）— ただし内容はユーザー由来
3. **KPI・滞在時間・人数目標は書かない**。質感・迷い・寂しさ・操作の重さなど
4. ユーザーの観察文は **要約せず核を残す**。言い換えは最小限
5. 日本語。concept のトーンに合わせ、箇条書きで具体的に

## セクションごとに誰が書くか

| セクション | 誰が書くか |
|---|---|
| メタ行（タグ・観察期間・位置づけ） | ユーザーと対話で確定 |
| この版の位置づけ | ユーザーと対話で確定（実装羅列にしない） |
| この版で試したこと | 事実は diff / コミットから整理可。**ユーザーに確認してから** 載せる |
| **観察** | **ユーザーだけ**（対話で聞いた内容） |
| **気づき** | **ユーザーと合意した内容だけ** |
| v次版 への問い | ユーザーが挙げた問い・仮説 |
| 次に試す | ユーザーと絞った 1〜3 件 |

---

## どのファイルを編集するか

| 状況 | 操作 |
|---|---|
| 版 `vX.Y.Z` の初回まとめ | 対話完了後に `docs/journal/vX.Y.Z.md` を新規作成 |
| 同じ版の追加観察 | 対話 → 確認 → 既存の「観察」「気づき」「次に試す」へ統合 |
| 手順・採番の変更 | `docs/journal/README.md`（journal 依頼でなければ触らない） |

ファイル名はタグと一致（`v0.2.0` → `v0.2.0.md`）。タグ未作成の版はメタ行に `（journal 確定後に付与）` と書いてよい。

## 記事テンプレート

対話で中身が揃ってから使う。プレースホルダーのまま推測で埋めない。

```markdown
# vX.Y.Z

- **タグ**: `vX.Y.Z`（未付与ならその旨）
- **本番**: https://breathing-4k7.pages.dev
- **観察期間**: YYYY-MM-DD 〜 …
- **位置づけ**: この版を一言で

## この版の位置づけ

（対話で確定した一文〜短い段落）

## この版で試したこと

### [領域名]

- （ユーザー確認済みの事実のみ）

## 観察

### [観察対象]

- **短い見出し。** ユーザーの体験の言葉

## 気づき

### [気づきのまとまり]

1. **一文の結論。** ユーザーと合意した解釈

## v次版 への問い

- （ユーザーが挙げた問い）

## 次に試す（v次版 へ）

- [ ] （ユーザーと絞った試すこと）
```

### 追記テンプレート

同じ版への追加も **対話 → 確認 → 統合**。時系列を残したい場合だけ末尾に:

```markdown
## 追記（YYYY-MM-DD）

### 観察

- （ユーザー口述）

### 気づき

8. **…**（既存番号の続き。合意済みのみ）

### v次版 への問い（実装前のメモ）

- （ユーザーが挙げた問い）
```

---

## 書き方のチェックリスト（ファイル更新前）

- [ ] `## 観察` の各行が、ユーザーがチャットで言ったことに対応している
- [ ] `## 気づき` はユーザーが確認・修正したあとの文だけ
- [ ] 実装差分からの推測文が観察・気づきに混ざっていない
- [ ] 版番号・観察期間がユーザーと合意している
- [ ] 「次に試す」は 3 件以内でユーザーと絞った
- [ ] 実装ファイルを変更していない（依頼が journal のみの場合）

## よくある依頼パターン

**`/write-journal` だけ（観察の口述なし）**
→ いきなり journal を書かない。版を確認し、「何を観察したか」「どんな質感だったか」を質問する。事実の整理（試したこと）が必要なら diff を提示し、観察とは分けて示す。

**ユーザーが観察を口述した**
→ 聞き漏れがないか短く確認。気づきは一緒に言語化し、草案を見せてから `## 観察` `## 気づき` に反映。

**版の締め・v0.X.Y としてまとめたい**
→ これまでの対話で出た観察・気づきだけを統合。会話にない段落は追加しない。git 手順は案内のみ（コミット・タグはユーザー依頼時）。

**journal だけ、実装を変えないで**
→ スキル全体のデフォルト。コード・CSS・docs/concept も触らない（journal 配下のみ）。

---

## コミット（ユーザーが依頼したときのみ）

対話で内容が確定し、ユーザーがコミットを依頼したときだけ。実装コミットとは別に:

```bash
git add docs/journal/vX.Y.Z.md
git commit -m "$(cat <<'EOF'
journal(vX.Y.Z): 一行サマリー

EOF
)"
```

タグ付与は [docs/journal/README.md](../../../docs/journal/README.md) §2 に従う。
