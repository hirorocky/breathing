# 探求の目的

## 何をしたいか

存在しやすい空間に必要な要素を、Web 上で探求する。

動くサイトを置き、そこに滞在し、空気を観察する。気づきを次の修正へ回す——**修正 → 観察 → 気づき → 次のバージョン**のループで、少しずつ近づいていく。完成形を目指すのではなく、問いを立てたまま試し続ける小さな探求である。

人が過剰に補正したり演じたりせず、比較的そのまま存在できる余白——「存在負荷を下げる空気」——が、どの設計から生まれるかを確かめる。

## 探求のループ

| 段階 | やること |
|---|---|
| 修正 | サイトの要素・振る舞い・間を変える |
| 観察 | 自分や他者がそこにいるとき、何が起きるか見る |
| 気づき | 存在負荷が下がったか、逆に上がったかを言葉にする |
| 次のバージョン | 気づきを設計判断へ戻し、また修正する |

正解を一度に当てるより、ループの回転数で近づく。

### git との対応づけ

| 段階 | 残し方 |
|---|---|
| 修正 | 通常コミット |
| 観測版の固定 | `package.json` の version + git tag `vX.Y.Z` |
| 気づき | `docs/journal/vX.Y.Z.md` + `journal(vX.Y.Z):` コミット |

タグは観察対象の版、journal はその版から得た気づき。セットで 1 サイクル。手順の詳細は [journal/README.md](../journal/README.md)。

- 補正量を下げる
- 存在コストを下げる
- 未完成な状態を許容する
- 過剰な自己演出を減らす

## 問題意識

多くのオンライン空間では、次が起きやすい。

- キャラクター化
- 承認獲得競争
- 即時反応の圧力
- 正しいコミュニケーションへの適応

その結果、思考途中の状態・曖昧さ・試行錯誤・本来のテンポが失われていく。

## 仮説

存在負荷の低い空間には、次の特徴があるのではないか。

- 少し黙っていてもよい
- 解像度が合っていなくてもよい
- 即座に価値を出さなくてもよい
- 未完成な言葉を置いておける
- キャラクターを固定されない

## 検証したいこと

- どの設計が存在負荷を下げるか
- どの機能が補正や自己演出を生むか
- 未完成な言葉を残せる場は成立するか
- 承認指標を減らしても場は維持できるか
- 人はいつ「存在しやすい」と感じるか

## 成功条件

利用者数や滞在時間では測らない。次の体験が生まれること。

- 少し補正せずにいられた
- 未完成なまま表現できた
- 存在コストが下がった

KPI ダッシュボードや成長指標は載せない。数値より、滞在の質感と設計の副作用を観察し、ループの次の一手にする。

## オンライン化の位置づけ

利用者を増やすことが目的ではない。不特定多数が同じ空間に重なったとき、設計が仮説を支えるかを観察するためのインフラを用意する。

オンラインで意図的に入れないもの:

| パターン | 負荷が上がる理由 |
|---|---|
| キャラクター化 | 一貫した自己演出が要求される |
| 承認獲得競争 | いいね・ランキングが補正を促す |
| 即時反応の圧力 | 投稿直後のフィードバック期待 |
| 正しいコミュニケーションへの適応 | スレッド・返信・既読 |
