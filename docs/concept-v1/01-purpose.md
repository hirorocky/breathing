# 探求の目的

**社会の中で、演じなくていい時間を、同じ部屋で試す。**

## 何をしたいか

**社会の中の存在負荷を、下げられるかを探求する。**

存在負荷が高い私たち——初対面でも、知り合いでも、同じ空間にいるとき——が、**StackChan**（[M5Stack のデスクトップロボット](https://docs.m5stack.com/ja/StackChan)）を部屋の中心に置き、存在負荷が高い状態から、ゆるやかに低い状態へ向かえる時間を試す。人数はおおよそ 2〜10 人を想定する。

完成形の「正しい付き合い方」を先に決めるのではなく、動く場を置き、同席して観察し、気づきを次の版へ回す——**修正 → 観察 → 気づき → 次のバージョン**のループで近づいていく。

## 媒体：Web ではなく StackChan

v0.1.0 / v0.2.0 は **Web 上の静かな場** を試した。v0.2.0 の観察から、一人の Web では他者感が薄れ、リアル同席には **画面を介さない容器** が要ると分かった。

**v1.0.0 以降**の探求媒体は **StackChan** である。

- **同じ部屋に physically いる** — 机の上の小さな身体。画面越しではない
- **共有リズムを身体で示せる** — サーボ（頭の上下）、LCD の表情・リング、**RGB LED（12 個）**
- **カスタムファームウェアで制御できる** — Arduino / UiFlow2 等（[StackChan ドキュメント](https://docs.m5stack.com/ja/StackChan)）
- **複数台・遠隔操作も選択肢** — ESP-NOW、**リモートコントローラー**（K151-R 等）

Web サイト（v0.1.0 / v0.2.0）は git 履歴に残す。**v1.0.0 以降の主たる実装・観察対象は StackChan**（`stack-chan/` サブモジュール + `overlay/`）。

## スタンス

- **人は要る。** 自己認識には他者の存在が必要である、という前提から始める
- **社会から逃げない。** 存在負荷を下げたいのは、孤立のためではなく、**関係の中**のため
- **StackChan は人の代わりではない。** 第三の焦点・共有リズムの容器。人間同士の関係を置き換えない
- **v0.1.0 型には戻らない。** 匿名の人数表示、常時入力、読める社会的 UI で他者を合成しない
- **出荷時の AI Agent モードはそのまま使わない。** 質問応答・会話エージェントは存在負荷を上げやすい。静かな呼吸ファームウェアを書く

## 探求のループ

| 段階 | やること |
|---|---|
| 修正 | ロボットの振る舞い・間・部屋での置き方を変える |
| 観察 | 同室で同席したとき、何が起きるか見る |
| 気づき | 存在負荷が下がったか、逆に上がったかを言葉にする |
| 次のバージョン | 気づきを設計判断へ戻し、また修正する |

正解を一度に当てるより、ループの回転数で近づく。

### git との対応づけ

| 段階 | 残し方 |
|---|---|
| 修正 | 通常コミット（ファームウェア・設定） |
| 観測版の固定 | version + git tag `vX.Y.Z` |
| 気づき | `docs/journal/vX.Y.Z.md` + `journal(vX.Y.Z):` コミット |

手順の詳細は [journal/README.md](../journal/README.md)。

- 補正量を下げる
- 存在コストを下げる
- 未完成な状態を許容する
- 過剰な自己演出を減らす

## 問題意識

存在負荷が高い私たちが、同じ空間にいるとき、次が起きやすい。

- 何を話すか決めなければならない
- 沈黙を埋める圧力
- 印象管理・正しい自己提示
- 相手の反応を読み続ける負荷

その結果、**一緒にいるはずなのに、演じ続けている**時間が増える。

v0.2.0 では、一人で Web に入ると操作は軽いが **他者がいない** ために輪郭が薄れた（「無」）。

StackChan の **デフォルト機能**（AI エージェント、音声 Q&A、チャット的な応答）も、同じ方向の負荷を生みうる。探求では意図的に避ける。

## 仮説

存在負荷を下げるには、機能の有無より **関係と時間の段階** が効くのではないか。

- 最初から言葉を置かせない
- 共有の呼吸でリズムを先に揃える — StackChan の頭・画面・光で示す
- 価値観・言葉のすり合わせは、後半にゆるやかに（口頭中心。チャット UI ではない）
- ロボットは他者を「表示」するのではなく、**一緒にいる時間の容器** になる
- 物理同席（人 + 机の上の StackChan）が、他者感と第三焦点の土台になる
- 存在負荷に **勾配** がある——高い状態から、ゆるやかに低い状態へ

## StackChan で使う／使わない（初期整理）

[StackChan のハードウェア](https://docs.m5stack.com/ja/StackChan)のうち、探求で向きそうなもの。**全部を最初から使う必要はない。** 観察しながら取捨選択する。

| 要素 | 向きそうな使い方 |
|---|---|
| サーボ（頭の上下・左右） | 呼吸リズムの共有。派手なジェスチャーは避ける |
| LCD（表情・アニメ） | 呼吸リングに相当する静かな視覚。説明テキストは最小 |
| **RGB LED（12 個）** | 吸／吐・フェーズ・昼夜の気配。明滅は弱く、強い演出は避ける |
| タッチ（頭頂） | 任意の最小操作。必須にしない |
| **リモートコントローラー**（K151-R 等） | 参加者が本体に触れず、テンポやフェーズを調整する **選択肢**。ESP-NOW 経由 |
| ESP-NOW | 複数台の同期、コントローラーからの遠隔操作（要検証） |

### RGB LED

本体リングの **12 個の RGB LED**（WS2812C）も利用候補に入れる。

- 呼吸周期と連動した明度・色のゆっくりした変化
- 勾配のフェーズ（前半／中盤／後半）の **低解像度な合図**
- v0.2.0 の昼夜・Orbs・星空の **身体側の気配** としての代替候補

常時点滅・強い色・ゲーム的なフィードバックは避ける。観察で、存在負荷を下げるか上げるかを見る。

### リモートコントローラー

[Remote Controller Kit（K151-R）](https://docs.m5stack.com/ja/StackChan) や ESP-NOW 送信側も、**使うかどうかは版ごとに試す選択肢** とする。

- 司会・設計者が、参加者の存在負荷を上げずにテンポを微調整する
- 複数 StackChan のグループ制御（同一 Receiver ID）
- 本体タッチより、**距離を保った操作**（触れない／触らせない場向け）

参加者全員にコントローラーを渡す想定ではない。操作が増えると存在負荷が上がる可能性があるため、journal で副作用も見る。

意図的に使わない／抑えるもの:

| 要素 | 避ける理由 |
|---|---|
| AI Agent（出荷時） | 会話・質問応答が存在負荷を上げる |
| 音声ウェイクワード | 能動的に話しかけさせる圧力 |
| スマホアプリ経由の Avatar | 画面を増やし、演じの対象を増やす |
| 常時マイク入力 | 聞かれている感・正しい発話の圧力 |

## 検証したいこと

- 同室に StackChan を置いたとき、一人の Web より存在負荷はどう変わるか
- 2〜10 人で、人数が増えると勾配はどう変わるか
- ロボットの呼吸（サーボ + 画面）で、高い存在負荷は下がるか
- 会話エージェントなしの静かなファームウェアで、第三焦点として足りるか
- **RGB LED** が昼夜・フェーズの気配として効くか、それとも邪魔か
- **リモートコントローラー** は存在負荷を下げるか（司会者用の距離操作として）
- 複数台（ESP-NOW）が要るか、1 台で足りるか
- 「存在負荷の勾配」は、設計で作るより **体感で生まれる** か

## 成功条件

人数・滞在時間では測らない。次の体験が生まれること。

- しばらく演じなくていられた
- 相手がそこにいる感覚があり、自分の輪郭も保てた
- 呼吸・価値観・言葉が、無理なくすり合わせられた
- 存在コストが下がった（操作ではなく、**関係の質**として）

KPI ダッシュボードや成長指標は載せない。数値より、同席の質感と設計の副作用を観察し、ループの次の一手にする。

## Web 探求からの転換（v0.2.0 → v1.0.0）

| Web（v0.1.0 / v0.2.0） | StackChan（v1.0.0 以降） |
|---|---|
| Web 上で存在しやすさを探す | **同室 + 机の上のロボット** |
| 画面の呼吸リング | サーボ + LCD の共有リズム |
| Orb・星空などの気配 | LED・表情など、身体に近い最小信号 |
| ブラウザ同期 | 不要（同一物理空間が前提）。ESP-NOW は補助 |
| 一人でも開ける | 人が要る。ロボット単体では完結しない |

v0.1.0 / v0.2.0 の journal は Web 前提の観察として残す。**v1.0.0** からはこの purpose を起点に、StackChan ファームウェアと同席観察を回す。
