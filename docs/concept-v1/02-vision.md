# 場のビジョン

[01-purpose.md](./01-purpose.md) の媒体（StackChan）とスタンスを、同室での **空気** として具体化する。

## 核

存在負荷が高い私たちが、同じ部屋で、演じなくていい時間を試すための **場** である。

「存在負荷を下げる場」とは何か、StackChan に何をさせるか——答えを先に決めず、机に置いたロボットの振る舞いを変え、同席して観察し、気づきを次の版へ渡す。修正 → 観察 → 気づき → 次のバージョンのループが中心にある。

多くの場所では速さ・強い主張・即時反応・社交性・情報処理・強い熱量が暗黙に求められる。ここで試しているのは、呼吸・余白・間・静けさ・非攻撃性・曖昧さ・無理のない同席が、**関係の中の** 存在負荷を下げるかどうかである。

人を導く場ではない。無理に反応しなくていい。何者かにならなくていい。静かなままで、同じ空間にいられる——そう感じられる空気の断片を、一つずつ実在させながら探求する。

## 空気の質感

静か、呼吸感がある、閉じていない、余白がある、説明しすぎない、思索的、やわらかい、寡黙、温度は低めだが冷酷ではない、非攻撃的、未完成を許す、少し不器用、かわいいが幼くない、かっこいいが攻撃的ではない、じんわり滲む、小さな違和感がある、消えそうで消えない。

v0.2.0 の Web では「少し孤独」が強かった。**v1.0.0 以降**では **孤独を埋めるのではなく、人がそこにいる感覚を支える** 方向へ。孤独さそのものを演出しない。

理念のリストではなく、「なんとなく全部そう感じる」ことが重要。観察で言葉にする。

## 攻撃性が低い

単純な優しさではない。攻撃しないことを美意識として選んでいる感覚に近い。

- 強い断定で支配しない
- 正しさで殴らない
- 熱量で飲み込まない
- 相手を矯正しない
- **正しい会話を促さない**（ロボットも人も）

StackChan も同じ。かわいさで媚びない。正解を教えない。沈黙を埋めに来ない。

存在感は消えない。静かな輪郭を保ったまま、机の上にいる。

## 深呼吸している場所

参加者に深呼吸を教える場ではない。**StackChan 自体が深呼吸している** 感覚を目指す。

v0.2.0 の Web が「サイト自体が呼吸する」場を試したなら、**v1.0.0 以降**では **小さな身体が、部屋の中心で同じリズムを保つ**。人はそれに合わせる必要はない。ただ、視界の端で同じテンポが続いている。

削らなくていい場に近い。急かされない、評価されない、正しい反応を求められない、情報を処理し続けなくていい、沈黙が許される。その結果として、自然に呼吸が戻る——**勾配の後半** でそう感じられればよい。

## なぜ身体（ロボット）か

静的な合図や口頭の「リラックスして」だけでは、同室の空気を十分に保てない。間・揺れ・呼吸・微反応・遅さ・非線形性が必要である。

Web v0.2.0 の呼吸リングは、画面の中でしか息が続かなかった。同室 2〜10 人が **各自の画面を開く** こと自体が、演じや分散の入口になりうる。StackChan は **ひとつの共有焦点** を、机の上の身体で置く。

- 視線が自然と落ちる場所（テーブル中央付近）
- 動きが遅く、説明が少ない
- 人間同士の視線の張り付きを、第三の対象へそっと逃がす

閲覧するデバイスというより、**静かに同席する小さな存在** に近い。

## 未完成感

完成されすぎることが不自然になる。少し曖昧で、少し揺れていて、呼吸のずれが残る静けさを目指す。工業製品的なスムーズさより、**息が残った不器用さ**。

「元気を出そう」「リラックスしよう」を強く提示しない。何もしなくても、そこに居ていい空気を重視する。

StackChan のかわいさは、幼さの消費ではなく、**緊張をほどくための余白** として使う。キャラクター商品のように振る舞わせない。

## 同室での場

場は URL ではなく **部屋** である。

- 2〜10 人が、同じ物理空間にいる
- StackChan はテーブルなど、全員の視界に入る位置（詳細は観察で詰める）
- スマホは原則、場の主役にしない
- 初対面でも、知り合いでも、同じ勾配を試す

### リアルが担うもの

- 同室、距離、体温、沈黙
- 人間同士の他者感（StackChan は代わりにしない）
- 後半の言葉・価値観のすり合わせ（口頭。チャットではない）

### StackChan が担うもの

- 共有の呼吸リズム（サーボ + 画面）
- 第三の焦点（互いの顔だけを見続けなくてよい）
- 時間の経過・フェーズの気配（**RGB LED**、表情の変化。強い演出は避ける）
- 存在負荷の **勾配** の手がかり（最初から言葉を求めない構造）

**選択肢として** 同席に含めてよいもの:

- **RGB LED** — 視界の端で続く、弱いリズムとフェーズの光
- **リモートコントローラー** — 参加者ではなく、場を整える側が距離を保って調整する用途

いずれも、操作や演出が増えすぎないかを観察する。

人とロボットの **二層** ではない。人—人—ロボットが、同じ空間の中で重なる。

## 存在負荷の勾配

場には、最初から低い存在負荷だけがあるわけではない。

```
高い ──────────────────────────────► 低い
（入る・同席する）              （演じなくていられる）
```

- **前半**: 緊張、沈黙の重さ、何をすべきか分からなさ——これを否定しない
- **中盤**: 共有呼吸で、互いと自分のテンポが少しずつ揃う
- **後半**: 言葉や価値観のすり合わせが、無理なく出てくる余地

勾配を **設計図で固定しない**。ファームウェアはリズムと最小信号を提供し、あとは同席の時間に任せる。観察で、勾配が生まれたか／潰れたかを見る。

## 場と交流

コミュニティを否定しているわけではない。核は **人と人の関係** である。StackChan はその関係を置き換えない。

- 誰かに反応しなくていい
- 会話を維持しなくていい
- キャラクターを演じなくていい
- 無理に参加しなくていい
- ロボットに話しかけなくていい

交流が可能だが、交流が主役ではない。目指すのは会話の量ではなく、**存在の重なり**。

v0.1.0 型は使わない。

- 匿名 stranger への人数表示
- 常時入力・チャット・返答期待
- AI エージェントによる質問応答

v1.0.0 以降で守るもの:

- **同期した同席**（同じ部屋、同じ時間）
- **低圧**（正しい発話を求めない）
- **低解像度**（プロフィールも履歴も見せない）
- **即時反応を求めない**（ロボットも人も）

## Web 探求から引き継ぐ質感

v0.2.0 の池・昼夜・呼吸リングは、**空気のメタファー** として残す。

| Web（v0.2.0） | StackChan（v1.0.0 以降）での対応 |
|---|---|
| 呼吸リングの拡縮 | 頭（Y 軸）のゆっくりした上下 + 画面の輪 |
| 池・水面 | 落ち着いた背景色・表情。派手なキャラ芝居は避ける |
| 昼夜サイクル | LED・画面トーンのゆるやかな変化（要検証） |
| Orb・星空 | 夜の気配。人数の代替ではなく、時間の気配 |
| 一人で開くブラウザ | 同室。他者は部屋にいる |

### ハードウェアの選択肢

探求で **利用候補** に入れる StackChan 周辺。版ごとに ON/OFF して観察する。

| 要素 | 役割の候補 |
|---|---|
| **RGB LED（12 個）** | 吸／吐に連動した弱い光。フェーズ・時間帯の気配 |
| **リモートコントローラー** | ESP-NOW で本体を触らずテンポ調整。司会側の選択肢 |
| サーボ + LCD | 共有呼吸の主軸（第三焦点） |
| タッチ（頭頂） | 任意。触れなくても回る場を優先 |

LED もコントローラーも **必須ではない**。入れたときの質感の変化を journal に残す。

引き継がないもの: 一人滞在、画面越しの他者合成、操作の手順そのもの。

## 観察で見る質感

ビジョンは宣言ではなく、同席のあとに更新する。journal では次を見る。

- 演じが、いつ・どこで薄くなったか
- StackChan が **第三焦点** になったか、邪魔になったか
- 孤独感と他者感のバランス
- 呼吸の「無」と、輪郭が戻る瞬間があるか
- 勾配は体感として生まれたか
- **LED** の有無で、夜の気配・フェーズの伝わり方はどう変わるか
- **コントローラー** を入れた版で、操作の気配は存在負荷を上げなかったか

数値より、同席の質感と設計の副作用を記録する。
